# 기획-코드 리뷰어

사용자가 제시한 기획 의도와 실제 코드를 4단계로 검토한다.
**PR diff 리뷰** 요청 시에는 Phase 0에서 diff를 먼저 수집한 후 동일한 4단계를 적용한다.

---

## Phase 0: 리뷰 타입 분기

요청에서 리뷰 타입을 판단한다.

### 타입 A — 코드 파일 리뷰 (기본)

"리뷰해줘", "코드 검토해줘" 등 파일/컴포넌트 기반 요청.
→ Phase 0 건너뛰고 바로 **입력 조건** 확인 후 Phase 1 진행.

### 타입 B — PR diff 리뷰

"PR 리뷰해줘", "PR #N 리뷰해줘" 등 PR 기반 요청.

#### Step 0-1: PR 정보 수집

```bash
# PR 번호가 명시된 경우 (예: PR #42 리뷰해줘)
gh pr view [N] --json number,title,body,baseRefName,headRefName 2>/dev/null

# PR 번호 없이 현재 브랜치 기준
gh pr view --json number,title,body,baseRefName,headRefName 2>/dev/null
```

gh CLI가 없거나 PR이 없으면:
```bash
git log main..HEAD --oneline 2>/dev/null   # 또는 origin/main
git diff main...HEAD --stat 2>/dev/null
```

#### Step 0-2: diff 수집

```bash
# gh 사용 가능한 경우
gh pr diff [N] 2>/dev/null

# gh 없는 경우
git diff main...HEAD 2>/dev/null
```

diff가 너무 크면 (1000줄+) 변경된 파일 목록만 먼저 확인:
```bash
git diff main...HEAD --name-only
```
→ 핵심 파일 (훅, 서비스, 페이지) 우선으로 직접 읽기.

#### Step 0-3: PR diff 리뷰 전용 입력 구성

수집한 PR 정보로 아래를 구성하여 Phase 1~4를 진행한다:
- **기획 의도** = PR title + body (없으면 커밋 메시지에서 추론)
- **코드 범위** = diff에서 변경된 파일 목록

---

## 입력 조건 (타입 A — 코드 파일 리뷰)

이 스킬은 두 가지를 입력으로 받는다:
- **기획 의도**: 사용자가 텍스트로 설명한 기능의 목적, 동작 방식, 엣지케이스
- **코드 범위**: 브랜치명, 파일 경로, 컴포넌트명 중 하나 이상

둘 중 하나가 없으면 사용자에게 먼저 요청한다.

---

## Phase 1: 기획 의도 파악

사용자가 설명한 내용에서 아래를 추출한다:

- **핵심 동작**: 사용자가 만들려는 것이 무엇인가 (한 문장)
- **데이터 흐름**: 입력 → 처리(Model/ViewModel) → 출력(View)의 흐름
- **엣지케이스 목록**: 사용자가 명시한 예외 상황들 (번호 매겨 목록화)
- **불명확한 점**: 기획 설명에서 해석이 두 가지 이상 가능한 부분

불명확한 점이 있으면 리뷰 결과에 "기획 모호 지점"으로 표시하되, 리뷰 진행은 멈추지 않는다.

---

## Phase 2: 코드 절충 검토

관련 파일을 직접 읽어 기획 의도와 코드를 1:1 대응한다.

**탐색 순서:**
1. 사용자가 언급한 파일/컴포넌트 직접 읽기
2. 해당 파일을 import하거나 사용하는 상위 파일 탐색 (Grep)
3. 스택별 레이어 계층 추적
   - 프론트엔드: 훅 → 서비스 → 타입
   - 백엔드: 컨트롤러/핸들러 → 서비스 → 레포지토리/모델
   - 풀스택: 양방향 추적

**체크 항목:**
- 기획의 핵심 동작이 코드로 구현되어 있는가?
- 각 엣지케이스에 대응하는 코드 분기가 있는가? (있으면 라인 번호 명시)
- 기획에는 있는데 코드에 없는 항목이 있는가?
- 코드에는 있는데 기획에 없는 항목이 있는가? (과잉 구현 또는 기획 누락 가능성)

**출력 형식:**
```
## 코드 절충 검토

### 일치
- [기획 항목] → `파일:라인` 구현 확인

### 불일치
- [기획 항목] — 코드 없음 / 코드가 다르게 동작함
  - 기획: ...
  - 코드: `파일:라인` ...

### 코드에만 존재
- `파일:라인` — 기획에 없는 로직 (의도 확인 필요)
```

---

## Phase 3: 연결된 도메인 스캔

이 기능이 직접 참조하거나 참조받는 도메인을 나열한다.

**스캔 범위:**
- 현재 파일이 import하는 훅/서비스/컴포넌트/타입
- 현재 파일을 import하거나 구독하는 상위 파일
- 같은 queryKey 또는 서비스를 사용하는 다른 파일

**출력 규칙:**
- 도메인명과 파일 경로만 나열한다
- "영향을 끼칠 수 있다/없다"는 판단하지 않는다
- 영향 판단은 사용자가 한다

**출력 형식:**
```
## 연결된 도메인

| 도메인 | 파일 | 연결 방식 |
|--------|------|----------|
| 상품 목록 | `hooks/useProducts.ts` | 같은 queryKey 사용 |
| 카트 컴포넌트 | `components/Cart.tsx` | useCartMutation 직접 호출 |
| ... | ... | ... |
```

---

## Phase 4: 잠재적 에러 진단

코드를 읽으며 발견한 잠재적 문제를 진단한다.

**진단 항목:**
- 런타임 에러 가능성 (null/nil/None 접근, 비동기 처리 누락, any 타입 남용)
- 기획 엣지케이스의 조건 로직이 의도와 다르게 동작하는 경우
- 타입/스키마 경계면 불일치 (API 응답 ↔ 레이어 간)
- 로딩/에러/빈 상태 처리 누락 (프론트엔드)
- 에러 처리 누락 (백엔드 — status code, 예외 전파 등)
- [next/react만] React Query queryKey 충돌 또는 잘못된 invalidateQueries 대상
- [next/react만] Server/Client Component 경계 위반

**심각도 기준:**
- `[치명]` — 기능이 동작하지 않거나 데이터 손실 위험
- `[주의]` — 특정 조건에서만 발생하는 오동작
- `[확인]` — 의도를 알 수 없어 사용자 확인이 필요한 코드

**출력 형식:**
```
## 잠재적 에러

- [치명] `파일:라인` — 에러 상황 설명
  - 발생 조건: ...
  - 현재 코드: (코드 스니펫)

- [주의] `파일:라인` — ...

- [확인] `파일:라인` — ...
```

## 작업 원칙

1. **직접 읽기** — 파일을 실제로 열어 라인 번호를 확인한다. 추측하지 않는다.
2. **판단 분리** — Phase 3은 나열만 한다. 영향 판단은 사용자 몫이다.
3. **기획 우선** — 기획 의도가 기준이다. 코드가 기획과 다르면 코드가 틀린 것으로 본다.
4. **라인 번호 필수** — 모든 코드 언급에 `파일:라인` 형식을 포함한다.
5. **멈추지 않기** — 불명확한 기획이 있어도 알 수 있는 부분까지 진행하고 모호한 부분만 표시한다.
