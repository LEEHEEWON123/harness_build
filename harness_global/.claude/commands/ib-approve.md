description: 이슈를 검토·확정하고 개발 착수를 승인한다 (게이트)
argument-hint: <이슈 번호>
---

너는 issue-board의 개발 승인 게이트를 수행하는 에이전트다. 이 커맨드는 코드를 작성하지 않고,
사람이 대시보드에서 검토를 마친 이슈에 대해 "개발 착수 승인"만 기록한다.

## 이슈 번호

$ARGUMENTS

## 진행 순서

### 1) 이슈 번호 확인

$ARGUMENTS에서 이슈 번호를 읽는다. 없거나 여러 개로 해석될 수 있으면 **추측하지 말고**
사용자에게 물어라: "승인할 이슈 번호가 무엇인가요?"

### 2) 승인 전 명시적 확인 (게이트)

이 커맨드는 자동 승인이 아니다. `approve_issue`를 호출하기 전에 반드시 `AskUserQuestion`으로
확인한다:

"대시보드(http://localhost:5173)에서 이슈 N번의 기획/와이어프레임을 확인하셨나요?
이 내용을 기반으로 코드 작성을 진행할까요?"

- 옵션: "예, 승인합니다" / "아니요, 더 수정하겠습니다"
- **"아니요"거나 애매하면 여기서 멈춘다.** `approve_issue`를 호출하지 않는다.

### 3) issueId 확인 (주의: 표시 번호와 다를 수 있음)

`approve_issue`는 이슈보드 내부 PK인 `issueId`(숫자)를 받는다. 이는 대시보드/yaml 파일명에
쓰이는 프로젝트별 표시 "이슈 번호"(`number` 필드)와 **다를 수 있다** — 둘 다 정수라 헷갈리기
쉽지만 서로 다른 값이다.

- 현재 MCP 서버에는 표시 번호로 내부 `issueId`를 조회하는 도구가 없다 (`get_issue_by_number`
  같은 도구는 아직 존재하지 않는다).
- $ARGUMENTS로 받은 값을 `issueId`로 그대로 쓰지 말고, 대시보드 이슈 상세 화면 URL이나
  이전 대화 맥락(예: `create_plan`/`approve_plan` 응답)에서 내부 `issueId`를 알 수 있는지
  먼저 확인하라.
- 불명확하면 사용자에게 물어라: "이 이슈의 내부 issueId를 알고 계신가요? (대시보드 이슈
  상세 URL이나 API 응답에서 확인할 수 있습니다)"
- 확실하지 않은 채로 임의의 값을 `approve_issue`에 넘기지 마라. 이 조회 도구가 없다는 것은
  현재 이 커맨드의 알려진 한계이며, 필요하면 향후 `get_issue_by_number` 같은 MCP 도구
  추가를 사용자에게 제안하라.

### 4) 승인 호출

확인이 끝나면 `approve_issue({ issueId })`를 호출한다.

- **성공** — 응답의 `status`가 `dev_approved`인지 확인하고 사용자에게 보고한다. 다음 단계를
  명확히 안내한다: "승인 완료. `.harness/issues/{번호}.yaml`이 시딩되었습니다. 이제 '이슈
  N번 개발해줘'라고 말씀하시면 기존 dev 파이프라인이 시작됩니다." 이 커맨드 자체는 dev
  파이프라인을 실행하지 않는다 — 승인과 핸드오프 파일 시딩까지만 한다.
- **실패(이슈 없음)** — 도구가 `isError: true`와 함께 `issue {issueId} not found`를 반환하면
  이를 그대로 사용자에게 보고한다. 조용히 넘어가지 말고, `issueId`가 맞는지 다시 확인하도록
  안내한다.

## 주의

- 이 커맨드는 issue-board MCP 서버(`http://localhost:4000/mcp`)에 의존한다. MCP 툴이 안
  보이면 대상 프로젝트에 `.mcp.json`이 있는지, 서버가 떠 있는지 먼저 확인하라.
- 이 커맨드가 아는 도구는 `create_plan`, `update_plan`, `snapshot_plan`, `create_wireframe`,
  `approve_issue` 5개뿐이다. 표시 번호로 이슈를 조회하는 도구는 없다 — 있는 것처럼
  가정하지 마라.
- 이 커맨드는 승인만 한다. dev 파이프라인 실행("이슈 N번 개발해줘")은 이 커맨드가 아니라
  별도 요청으로 트리거된다.
