# 레벨 자동 추론 설계

**날짜:** 2026-06-26  
**상태:** 확정

---

## 목표

사용자가 `low:` / `mid:` / `high:` 키워드를 명시하지 않아도, 요청 텍스트와 코드베이스 컨텍스트를 분석해 기획 레벨을 자동 결정한다.

---

## 접근 방식: 하이브리드 (Phase 0 텍스트 + Phase 1 코드 보정)

### Phase 0 — 텍스트 추론

오케스트레이터가 요청 텍스트를 분석해 예비 레벨 결정.

**추론 기준표:**

| 신호 키워드/패턴 | 판정 |
|----------------|------|
| "텍스트", "색", "스타일", "오타", 단일 속성 변경 | `low` 확실 |
| 단일 컴포넌트 / 훅 / API 엔드포인트 명시 | `mid` 확실 |
| "페이지", "플로우", "전체", "모듈", "시스템", "여러 ~" | `high` 확실 |
| "버그", "에러", "고쳐", 범위 불명확한 "기능" | 애매 → 사용자 질문 |

**출력 형식:**

```
# 명확 케이스
[레벨 추론: mid → sonnet | 신규 훅+서비스]

# 애매 케이스
범위가 명확하지 않아 레벨을 판단하기 어렵습니다.
low (단순 수정) / mid (기능 1~2개) / high (신규 페이지·모듈) 중 어느 레벨인가요?
```

- 명확 케이스 → 추론 레벨로 Phase 1 진행 (중단점 없음)
- 애매 케이스 → 사용자 답변 대기 후 진행

---

### Phase 1 — code-analyzer 재추론 (보정)

code-analyzer가 코드베이스 스캔 후 `01_spec.md`에 `INFERRED_LEVEL` 포함.

```markdown
## 추론 레벨
- INFERRED_LEVEL: mid
- 근거: 기존 useAuth 훅 존재, 신규 서비스 1개 + 컴포넌트 1개 필요
```

오케스트레이터 비교 로직:

| Phase 0 | Phase 1 | 처리 |
|---------|---------|------|
| mid | mid | 조용히 유지 (출력 없음) |
| mid | high | `[레벨 조정: mid → high \| 기존 auth 없음, 신규 레이어 3개]` 출력 후 레벨 변경 |
| 질문으로 확정된 값 | 다른 값 | Phase 1 추론 무시 (사용자 명시 우선) |
