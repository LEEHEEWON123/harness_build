# 기획 라운드(N차기획) & 이슈 차수 필터 설계 스펙

**날짜:** 2026-07-15
**상태:** 승인됨

---

## 배경

issue-board는 현재 프로젝트당 기획을 사실상 1개로 취급한다. `plans` 테이블 자체는 `project_id`만 갖고 유일성 제약이 없어 여러 기획을 이미 저장할 수 있지만, 대시보드(`plan/page.tsx`)는 이슈에 연결된 기획 또는 가장 최근 기획 하나만 보여주고, 여러 기획을 나열/전환하는 UI가 없다. 이슈 탭도 프로젝트의 전체 이슈를 기획 구분 없이 한 목록으로만 보여준다.

실제 사용 흐름은 `1차기획 → 개발 완료 → 2차기획 → 개발 완료 → ...`처럼 순차적으로 반복된다. 개발이 끝난 뒤 새 기능을 다시 기획하면 이슈가 계속 누적되는데, 어느 라운드에서 나온 이슈인지 구분할 방법이 없어 관리가 어렵다.

---

## 목표

- 프로젝트당 여러 기획(라운드)을 만들고, 대시보드에서 라운드별로 전환해 볼 수 있게 한다
- 이슈 탭에서 "몇 차기획에서 나온 이슈인지" 필터링할 수 있게 한다
- `/ib-plan`이 새 라운드를 시작할 때 직전 라운드의 진행 상태와 "범위 밖" 항목을 참고 자료로 활용하게 한다

## 범위 밖 (이번 스펙에 포함하지 않음)

- 라운드 진입/전환에 대한 시스템 차원의 강제(gating) — 직전 라운드 이슈가 안 끝나도 새 라운드 생성은 항상 허용. `/ib-plan`이 안내만 한다
- 기획(Plan) 삭제 기능 — 현재도 없고 이번 스펙에서도 추가하지 않음
- 차수를 사용자가 직접 지정/재배열하는 기능 — 차수는 항상 생성 순서로 계산

---

## 설계 원칙: 차수는 저장하지 않고 계산한다

`plans` 테이블에 "차수" 필드를 추가하지 않는다. 대신 프로젝트의 `plans`를 `id ASC`(생성 순서)로 정렬했을 때의 순번을 그대로 "N차기획"으로 사용한다 — 1번째 행 = 1차기획, 2번째 행 = 2차기획.

이유:
- 스키마 마이그레이션이 필요 없다
- 저장된 "차수" 값과 실제 생성 순서가 어긋나는 데이터 정합성 문제가 원천적으로 없다
- 기획 삭제 기능이 없으므로 중간 라운드가 빠지면서 번호가 밀리는 경우도 발생하지 않는다

---

## 데이터 흐름

### 백엔드 (`apps/issue-board-mcp`)

- 신규 모델 함수 `listPlansByProject(db, projectId)` (`src/models/plans.ts`) — 해당 프로젝트의 모든 `Plan`을 `id ASC`로 반환
- 신규 REST 엔드포인트 `GET /api/projects/:projectId/plans` — 위 함수 호출, 대시보드 전용 (MCP 툴 추가는 불필요)
- MCP `get_project_context`는 이미 프로젝트의 전체 `plans`(id DESC)와 `issues`를 반환하므로, `/ib-plan`이 새 라운드를 시작할 때 필요한 정보(직전 라운드가 몇 차인지, 그 라운드 이슈들의 완료 상태, 직전 기획의 "범위 밖" 섹션)는 이 응답만으로 충분하다. MCP 서버 코드 변경 없음.

### 프론트엔드 (`apps/issue-board`)

- `lib/api.ts`에 `fetchPlans(projectId): Promise<Plan[]>` 추가 (id ASC 순서 유지 — 배열 인덱스 + 1이 곧 차수)

**기획(Plan) 탭 (`plan/page.tsx`, `PlanView.tsx` 주변):**
- `fetchPlans(projectId)`로 전체 라운드를 가져와 라운드 선택 UI(칩/탭)를 렌더링: `1차기획` `2차기획` ...
- 각 칩에 상태 배지 표시: `초안` / `확정`, 확정된 라운드는 진행률도 함께 — `이슈 3/5 완료` (완료 = 해당 planId를 가진 이슈 중 `dev_approved` 또는 `done` 개수 / 전체 개수)
- 기본 선택은 가장 최근 라운드(기존 `fetchLatestPlan` 동작과 동일한 결과)
- 칩 클릭 시 기존과 동일하게 `?planId=` 쿼리로 전환, `PlanView`는 변경 없이 재사용
- 라운드가 1개뿐인 프로젝트는 스위처를 숨겨 기존 화면과 동일하게 유지

**이슈(Issue) 탭 (`issues/page.tsx`, `IssueList.tsx`):**
- `issues/page.tsx`가 `fetchIssues`와 함께 `fetchPlans(projectId)`도 가져와 `IssueList`에 전달
- `IssueList`(기존에도 `'use client'`)에 차수 필터 드롭다운(select) 추가: `전체` / `1차기획` / `2차기획` ... — 로컬 state로 필터링, 페이지 새로고침 없음. 기본값 `전체`
- `전체` 상태에서도 어느 라운드 이슈인지 바로 구분되도록, 각 이슈 카드에 소속 차수 배지(예: `1차`) 표시

### `/ib-plan.md` 커맨드 변경

"보드에 적재" 단계에서 `get_project_context` 응답의 `plans` 배열을 참고하도록 안내를 추가한다. 기존 라운드가 있는 경우(`plans.length > 0`):
1. 가장 최근 라운드(`plans[0]`)에 연결된 이슈 중 `dev_approved`/`done`이 아닌 것이 있으면, 새 라운드 생성을 막지 않되 "직전 라운드가 아직 진행 중인데 새 라운드를 시작할까요?"라고 사용자에게 안내한다
2. 가장 최근 라운드의 "범위 밖" 섹션(`sections.outOfScope` — `parse-plan-markdown.ts`가 `## 4. 범위 밖`에서 항상 이 필드로 파싱해두므로 markdown 원문을 다시 긁을 필요 없음) 항목을 추출해, 새 기획의 "명확화 질문" 단계에서 후보로 제시한다 (예: "1차 범위 밖에 있던 A, B, C 중 이번에 포함할 게 있나요?"). 반드시 포함해야 하는 것은 아니며 참고 자료일 뿐이다
3. 새 라운드는 기존 "새 기획" 경로(`create_plan`)를 그대로 사용한다. title에 차수를 명시할 필요 없음 — 대시보드가 자동 계산한다

## 에러 처리

- `fetchPlans` 실패 시 기획/이슈 탭 모두 기존 `ConnectionErrorBanner`로 폴백 (다른 REST 호출과 동일한 패턴)
- 라운드가 0개(기획이 아예 없는 프로젝트)인 경우는 기존 "기획이 아직 없습니다" 안내를 그대로 유지

## 테스트

- 백엔드: `listPlansByProject`가 여러 기획을 `id ASC`로 반환하는지 단위 테스트, `GET /api/projects/:projectId/plans` REST 스모크 테스트
- 프론트엔드: `fetchPlans` 클라이언트 함수 테스트. 라운드 스위처/이슈 필터의 브라우저 동작은 `/run` 스킬로 로컬에서 수동 확인 (기존 관례상 이 앱은 E2E를 스펙 범위에 넣지 않음)
