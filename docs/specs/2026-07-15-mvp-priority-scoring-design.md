# 기획 MVP 우선순위 점수제 알고리즘 설계 스펙

**날짜:** 2026-07-15
**상태:** 승인됨

---

## 배경

`/ib-plan`이 기획서의 "핵심 기능 (MVP)" 표를 쓸 때, 각 기능의 우선순위(`높음`/`보통`/`낮음`)를 지금까지는 에이전트가 별도 기준 없이 감으로 정했다. 사용자가 8개 판단 기준표(사용자 가치, 핵심 흐름 연결성, 운영 가능성, 데이터·정책 기반, 리스크, 개발 난이도, 검증 효과, 대체 가능성)를 제시하며 이를 알고리즘화하고 싶어했다.

두 가지 방식을 검토했다:
- **숫자 계산식**: 8개 축에 사람이 점수를 입력하고 가중합. → 결국 그 점수도 에이전트가 감으로 채우게 되어, 산수만 추가되고 정확도는 오르지 않음. 기각.
- **에이전트가 8개 기준으로 직접 채점 + 근거 기록**: 에이전트(시니어 프로덕트 기획자 역할)가 각 기능을 8개 기준으로 스스로 채점하고, 그 근거를 기획서에 남긴다. 채택.

## 목표

- `/ib-plan`이 MVP 기능마다 8개 기준으로 점수를 매겨 우선순위(`높음`/`보통`/`낮음`)를 결정하게 한다
- 왜 그 우선순위인지 근거를 기획서에서 바로 확인할 수 있게 한다 (감으로 정한 게 아니라는 걸 보여줌)
- **애플리케이션 코드는 건드리지 않는다** — `/ib-plan.md` 프롬프트 수정만으로 구현한다

## 범위 밖

- 이미 승인된 과거 라운드의 우선순위를 소급 재채점하는 기능
- 8개 기준 자체의 가중치를 사용자가 라운드마다 커스터마이징하는 UI/기능 (지금은 모두 동일 가중치로 고정)
- 우선순위 산정 결과를 별도 DB 필드로 구조화 저장 (아래 "왜 코드 변경이 필요 없는가" 참고 — 근거는 기획서 마크다운 텍스트로만 남는다)

---

## 왜 코드 변경이 필요 없는가

MVP 표는 `/ib-plan`이 마크다운 전문(`content`)으로 작성해 `create_plan`/`update_plan`에 그대로 넘기고, 대시보드(`PlanView.tsx`)는 그 마크다운을 범용 테이블 렌더러로 그대로 그린다 — 컬럼 개수를 하드코딩하지 않는다.

서버 쪽 파서(`apps/issue-board-mcp/src/models/parse-plan-markdown.ts`의 `parseMvpTable`)는 표에서 **앞 3개 컬럼(우선순위/기능/설명)만** 구조화 필드로 뽑아 이슈 생성에 쓰고, 4번째 이후 컬럼은 `cells.length < 3` 체크만 하고 구조 분해 할당(`const [priorityRaw, title, description] = cells`)에서 자동으로 버려진다 — 에러도 안 나고 무시될 뿐이다.

따라서 MVP 표에 **"근거" 컬럼을 하나 추가**하면:
- 대시보드에는 그 컬럼이 그대로 보이고 (범용 렌더러)
- 이슈 생성 로직(우선순위/제목/설명)에는 영향이 없다 (파서가 무시)

→ `/ib-plan.md`의 §3 표 템플릿과 채점 지침만 수정하면 끝난다.

## 채점 알고리즘

`/ib-plan`이 MVP 기능표의 각 행(기능)마다 아래 8개 기준을 **0/1/2점**으로 채점한다 (동일 가중치, 합산 0~16점):

| 판단 기준 | 질문 | 2점(명확히 해당) 기준 |
|---|---|---|
| 사용자 가치 | 사용자의 핵심 문제를 해결하는가? | 없으면 사용자가 핵심 행동을 완료하기 어렵다 |
| 핵심 흐름 연결성 | 핵심 사용자 흐름 안에 있는가? | 없으면 탐색·신청·확인·수강·결과 확인 중 하나가 끊긴다 |
| 운영 가능성 | 운영자가 실제 업무를 처리하는 데 필요한가? | 없으면 운영자가 엑셀·수작업으로 우회해야 한다 |
| 데이터·정책 기반 | 나중에 구조를 바꾸기 어려운 기반인가? | 상태값·권한·이력·데이터 관계처럼 초기 설계가 중요하다 |
| 리스크 | 기술·보안·권한·외부 연동 리스크가 큰가? | 늦게 확인하면 전체 일정·구조에 큰 영향을 준다 |
| 개발 난이도 | 일정 안에 구현 가능한가? | 어렵더라도 핵심이면 단순 버전으로 포함을 검토한다 |
| 검증 효과 | 출시 후 중요한 학습을 줄 수 있는가? | 사용자 행동·운영 가능성을 확인할 데이터를 준다 |
| 대체 가능성 | 수동 운영·단순 방식으로 대체 가능한가? | 대체 불가능하면 우선순위가 높다 |

**등급 매핑:**
- 11~16점 → `높음`
- 5~10점 → `보통`
- 0~4점 → `낮음`

**근거 표기 규칙:** "근거" 컬럼에는 8개 기준을 전부 나열하지 않는다. 점수를 가장 크게 끌어올리거나 내린 1~2개 기준만 압축해서 적고, 괄호로 `(합산점수/16)`를 함께 남긴다. 예: `핵심흐름 필수·대체불가 (13/16)`.

## `/ib-plan.md` 반영 방식

"### 3) 기획서 작성" 섹션의 MVP 표 템플릿(현재 `| 우선순위 | 기능 | 설명 |`)을 4컬럼으로 확장:

```
## 3. 핵심 기능 (MVP)
| 우선순위 | 기능 | 설명 | 근거 |
| --- | --- | --- | --- |
```

표 작성 지침 바로 아래에 위 채점 알고리즘(8개 기준, 0/1/2점, 등급 매핑, 근거 표기 규칙)을 추가한다. 우선순위 값 자체는 지금처럼 `높음`/`보통`/`낮음` 중 하나로만 적되, **감이 아니라 이 채점을 거쳐서** 정하도록 지침을 바꾼다.

---

## 데이터 흐름

1. `/ib-plan`이 기획서를 쓸 때 각 MVP 기능을 8개 기준으로 채점 (에이전트 내부 추론, DB에 별도 저장 안 함)
2. 4컬럼 마크다운 표로 `content`에 포함해 `create_plan`/`update_plan` 호출
3. 서버는 기존과 동일하게 앞 3컬럼만 파싱해 이슈 생성 — "근거" 컬럼은 마크다운 원문(`sections.markdown`)에만 남고 구조화 데이터로는 저장되지 않는다
4. 대시보드는 `sections.markdown`을 그대로 렌더 — 4컬럼 표가 자동으로 보인다

## 에러 처리

- 없음 — 이 변경은 프롬프트 지침 수정뿐이라 런타임 에러 케이스가 새로 생기지 않는다. 기존 `parseMvpTable`은 이미 3컬럼 미만인 행을 건너뛰는 방어 로직이 있고(`cells.length < 3`), 4컬럼 이상도 그대로 안전하게 처리한다.

## 테스트

- 코드 변경이 없으므로 자동화 테스트 추가 대상 없음
- 수동 검증: `/ib-plan`을 실제로 실행해 4컬럼 표가 포함된 기획서가 생성되는지, `create_plan` 호출 후 대시보드 기획 탭에서 "근거" 컬럼이 정상 렌더되는지, 이슈 탭에서 이슈가 (우선순위/제목/설명만 반영되고 근거 컬럼 영향 없이) 정상 생성되는지 확인
- 기존 `parse-plan-markdown.test.ts`가 3컬럼 파싱을 이미 커버하고 있어 회귀 위험 낮음 — 4컬럼 케이스에 대한 신규 단위 테스트는 선택 사항 (코드 변경이 없어 굳이 필요하지 않음)
